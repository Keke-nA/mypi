# mypi 目标文档

## 1. 项目目标

`mypi` 的目标不是先做一个零散的 CLI，而是先建立一套可扩展的、可测试的、分层清晰的 TypeScript coding-agent 基础设施。

最终形态上，它应该是一个类似 Claude Code / Codex CLI 的终端工具，但实现路径上优先保证下面几件事：

- 不同模型厂商接入方式统一
- Agent 执行过程具备明确状态流转
- 终端展示层与核心逻辑解耦
- 上层 coding-agent 具备完整上下文和 session 管理能力

当前阶段的重点不是把所有能力一次做满，而是先把四层架构的边界定义清楚，让后续实现可以持续演进而不相互污染。

## 2. 设计原则

- 分层明确：每一层只解决本层问题，不把上层语义提前下沉，也不把底层细节向上泄漏。
- 协议优先：优先抽象统一输入输出协议，再做具体厂商和运行时适配。
- 状态可追踪：Agent 运行必须是可观察的，状态切换、工具调用、结束条件都要能被外部理解。
- 终端无业务：TUI 负责展示和交互，不负责决定 Agent 如何思考或调度。
- 可替换性：Provider、模型、TUI 框架、上层产品形态都应可以替换，不影响整体结构。
- 可测试性：每层都应有独立测试空间，优先保证核心协议、状态机和编排逻辑可以单测。

## 3. 四层架构

### 3.1 AI 层

AI 层用于屏蔽不同厂商之间的 API 差异，对上提供统一的大模型调用方式。

这一层的目标是形成：

- 统一协议
- Provider 路由
- 厂商适配器
- 模型目录
- 认证能力

对外使用方式保持尽量简单，核心调用形态为：

- `stream(model, context, options)`
- `complete(model, context, options)`

其中：

- `model` 表示目标模型或模型描述
- `context` 表示本次输入上下文
- `options` 表示调用参数、流式配置、推理配置或运行时附加选项

AI 层需要解决的核心问题：

- 统一不同厂商在消息格式、工具格式、流式事件格式上的差异
- 将模型选择与具体厂商实现解耦
- 提供稳定的鉴权与配置加载机制
- 输出统一的结果对象或统一的流式事件

AI 层当前不负责：

- Agent 循环
- 任务状态机
- 会话树管理
- 终端展示

换句话说，AI 层只负责“如何稳定地调用模型”，不负责“为什么调用、何时调用、调用后做什么”。

### 3.2 Agent 层

Agent 层是状态化的 Agent 编排层，负责一次完整任务执行的循环控制。

这一层的核心职责是：

- `state`
- `turn loop`
- `tool orchestration`
- `event lifecycle`

它解决的问题是：

- 当前任务是否应继续推进
- 当前是否应该调用工具
- 工具返回后是否继续进入下一轮推理
- 当前 run 是否已经完成

Agent 层需要具备接近状态机的循环流转能力，同时通过 event 向外暴露运行时边界信息，方便上层做 UI 展示、日志记录、调试或外部集成。

对外接口方向上，暂定为：

- `prompt()`：发起一次新的 run
- `continue()`：在已有上下文上追加一次 run

这里的关键语义是：

- 一个 `turn` 是一次完整的调用闭环
- 一个 run 可以包含多个内部步骤
- Agent 层负责把模型调用、工具调用、状态推进组织成一条完整链路

Agent 层依赖 AI 层，但不应该依赖具体 TUI。它的输出应该是状态变化、结构化结果和事件，而不是直接打印终端内容。

### 3.3 TUI 层

TUI 层负责终端运行时的展示与交互。

它的职责重点是：

- 将流式输出、状态变化、工具调用过程渲染到终端
- 提供用户输入与运行反馈的终端交互体验
- 在不侵入核心逻辑的前提下，表达一个 coding-agent 的运行过程

当前 TUI 框架还未确定，因此这一层现阶段更适合先定义目标，而不是绑定具体实现。

TUI 层的基本要求是：

- 能消费 Agent / coding-agent 发出的事件
- 能展示流式文本、阶段状态、错误信息和工具执行过程
- 能支撑后续更复杂的 session 展示能力

TUI 层当前不应承担：

- Agent 调度决策
- Provider 选择
- 上下文裁剪与注入策略

它本质上是“运行时交互界面”，而不是“业务编排中心”。

### 3.4 Coding-Agent 层

Coding-Agent 层是最终的产品装配层。它组合 AI 层、Agent 层和 TUI 层，并负责 coding-agent 真实场景中的上下文管理。

这一层当前最重要的方向是：

- 使用 AI 层完成统一模型能力接入
- 使用 Agent 层完成一次任务的执行编排
- 使用 TUI 层完成终端交互
- 引入 session tree，管理完整上下文演化

这一层完成后，项目才真正具备类似 Codex CLI / Claude Code 的产品雏形。

Coding-Agent 层后续大概率会承接的能力包括：

- session tree
- 对话历史与分支管理
- 工作区上下文采集
- coding 任务相关工具集成
- 配置、运行模式、命令入口

但现阶段，这一层仍然应保持“装配层”定位，不宜过早把底层抽象重新耦合在一起。

## 4. 分层依赖关系

推荐依赖方向如下：

- `package/ai`：最底层能力，不依赖其它业务层
- `package/agent`：依赖 `package/ai`
- `package/tui`：消费上层事件协议，不直接承载核心业务逻辑
- `package/coding-agent`：组合 `package/ai`、`package/agent`、`package/tui`

整体上希望形成如下关系：

```text
coding-agent
 ├── agent
 │    └── ai
 ├── tui
 └── session/context management
```

关键约束：

- AI 层不能感知 Agent 状态机
- Agent 层不能耦合终端细节
- TUI 层不能反向定义核心协议
- Coding-Agent 层负责装配，而不是替代下层职责

## 5. 目录规划

当前建议的项目目录按 monorepo 方式组织，先按你定义的方向落：

```text
package/
  ai/
    src/
    test/
  agent/
    src/
    test/
  tui/
    src/
    test/
  coding-agent/
    src/
    test/
docs/
  goal.md
```

其中：

- `src/` 放实现代码
- `test/` 放该层级对应的测试代码

后续通常还会补充：

- 根目录 workspace 配置
- TypeScript 基础配置
- lint / format / test 配置
- 共享类型或工具包

但这些属于工程化细节，不影响当前目标文档的主方向。

## 6. 当前阶段的里程碑建议

### 阶段一：打底层协议

优先完成 AI 层最小可用版本，目标是跑通统一的 `stream` / `complete` 调用路径，并明确模型描述、上下文结构和统一返回格式。

### 阶段二：打通 Agent 闭环

在 AI 层之上实现 Agent 层最小闭环，至少具备：

- `prompt()`
- `continue()`
- 基本状态推进
- 工具调用插槽
- 事件生命周期输出

### 阶段三：补齐终端体验

实现 TUI 的 MVP，让一次完整 Agent run 可以在终端中可视化运行，包括流式输出、阶段状态和错误呈现。

### 阶段四：形成 coding-agent 产品雏形

在 Coding-Agent 层把 session tree、上下文管理、终端交互和任务执行真正串起来，形成一个可用的 coding-agent CLI。

## 7. 当前非目标

为了避免一开始范围失控，以下内容不建议在第一阶段作为重点目标：

- 一次性支持过多模型厂商
- 过早抽象复杂多 Agent 协作
- 过早绑定具体 TUI 框架实现细节
- 在 session tree 方案未清晰前，把上下文管理做得过重
- 先做 GUI 或 Web 端形态

当前最重要的是把四层边界做实，把最小闭环跑通。

## 8. 一句话总结

`mypi` 的目标，是先搭出一个分层清晰的 TypeScript coding-agent 架构，再在这个架构上逐步长出类似 Claude Code / Codex CLI 的完整产品能力。
