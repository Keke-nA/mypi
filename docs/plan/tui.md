# TUI 层细化计划

## 1. 目标

TUI 层的目标是在终端里稳定展示 coding-agent 的运行过程，并提供最小必要的用户输入能力。

这一层优先保证“可观察”和“可替换”，而不是一开始就绑定某个具体框架。

## 2. 职责边界

### 本层负责

- 消费标准化事件并渲染到终端
- 管理最小必要的终端输入交互
- 呈现流式输出、工具执行状态和错误信息
- 呈现 turn 边界和 session 提示

### 本层不负责

- 模型调用
- Agent 状态推进
- session tree 持久化
- 工作区工具实现
- provider 选择

### 上下游关系

- 上游是 `Coding-Agent` 层和 `Agent` 层发出的标准化事件
- 下游是具体终端输出介质和未来可能接入的 TUI 框架
- 本层不能直接依赖 provider SDK，也不能反向定义 Agent 协议

## 3. 公开接口

### `createTerminalUI(config)`

创建终端 UI 实例，内部装配 renderer、输入读取器和事件消费器。

### `render(event)`

消费标准化 `UiEvent` 并渲染到终端。

### `requestInput()`

向终端发起一次用户输入请求，用于 prompt 输入或恢复分支时的选择。

### `dispose()`

释放终端资源，结束渲染和输入监听。

## 4. 关键类型

- `UiEvent`
  终端层消费的标准化事件，统一文本流、工具状态、错误、turn 边界和分支提示。
- `TerminalUIConfig`
  TUI 的初始化配置，承载输出流、输入流、渲染偏好和交互开关。
- `Renderer`
  渲染器抽象，允许后续替换为不同的终端框架实现。
- `InputRequest`
  输入请求对象，描述 prompt、确认、选择等终端交互需求。
- `InputResult`
  输入请求结果，返回用户输入内容或终止状态。

## 5. 运行流程

1. 上层通过 `createTerminalUI(config)` 创建终端 UI。
2. `Coding-Agent` 或 `Agent` 将标准化事件转换为 `UiEvent` 并调用 `render(event)`。
3. TUI 按事件类型渲染流式文本、状态切换、工具执行和错误信息。
4. 当运行过程需要用户输入时，上层调用 `requestInput()`。
5. TUI 采集输入结果并返回给上层，自己不做任何业务决策。
6. 运行结束后调用 `dispose()` 释放资源。

## 7. 测试与验收

### 单测重点

- 文本流增量渲染连续且不打断
- 工具执行状态变化被正确展示
- 错误事件被正确分类展示
- 输入请求和返回结果的基本行为
- 长输出和工具输出的折叠策略

### 集成验收场景

- 一次普通文本回答能在终端持续流式显示。
- 一次工具调用过程能展示工具开始、执行中和结束状态。
- 从历史 turn 恢复时，终端能展示 branch 切换和摘要注入提示。

### 验收标准

- 终端上能稳定看到流式文本、工具执行状态、错误信息和 turn 边界。
- `TUI` 不依赖具体 provider，也不持有业务状态机。
- 替换 renderer 不需要修改上层业务协议。
